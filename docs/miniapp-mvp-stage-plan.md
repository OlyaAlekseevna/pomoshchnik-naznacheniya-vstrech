## Этап 2. User-flow parity

Статус: `done`

Цель:

1. Довести пользовательский путь до полного соответствия текущему bot-flow в рамках MVP.

План работ:

1. Проверить/доработать путь `Записаться` (длительность, неделя, дата, слоты, создание заявки).
2. Проверить/доработать `Мои заявки` (список, изменение цели, отмена).
3. Проверить корректность блоков `Профиль`, `Поддержка`, `Уведомления`.
4. Прогнать ручной e2e-сценарий пользователя.

Промежуточно уже проверено:

1. API-сценарий `auth -> booking -> create request -> list -> update goal -> cancel` проходит успешно.
2. Обнаружен ожидаемый конкурентный кейс: слот может стать недоступным между выбором и отправкой заявки.
3. В интерфейсе реализован понятный UX для кейса недоступного слота: сообщение + автообновление списка слотов без потери введенных данных формы.
4. Добавлен рабочий пользовательский блок оформления заявки (дата/слоты + контактные данные + согласие + отправка).
5. Добавлен список заявок пользователя с действиями `изменить цель` и `отменить` для редактируемых статусов.

Критерий завершения:

1. Сквозной пользовательский сценарий проходит в Mini App end-to-end.

## Этап 4. Hardening и безопасность

Статус: `done`

Цель:

1. Закрыть риски авторизации и доступов перед финальной приемкой.

Промежуточно уже проверено:

1. Dev-login дополнительно ограничен окружением: разрешен только в non-prod контурах (`dev/development/local/test`) и только при включенном флаге.
2. Негативные проверки auth/session:
   - запрос без Bearer-токена возвращает `401`;
   - просроченная Mini App session возвращает `401`.
3. Негативная проверка ролей:
   - доступ к admin endpoints с ролью `user` возвращает `403`.
4. Добавлены и пройдены тесты Mini App hardening: `tests/test_miniapp_security.py`.

Критерий завершения:

1. Проходят негативные проверки auth/roles, dev-login ограничен только `dev`.
