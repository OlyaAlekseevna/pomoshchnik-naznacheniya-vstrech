# ТЗ: Mini App Telegram (MVP) как расширение текущего бота

Дата фиксации: 31.05.2026  
Рабочая ветка: `dev`  
Статус: согласованный план этапов для продолжения разработки

## 1. Цель и рамки

Цель: реализовать Mini App в Telegram как дополнительный интерфейс к уже работающему боту согласования встреч с Google Calendar, без регрессий текущего bot-flow.

Обязательные ограничения:

1. Разработка только в ветке `dev`.
2. `main` используется только для продакшена, изменения попадают через merge после проверки.
3. Текущий Telegram-бот остается полностью рабочим параллельным каналом на всех этапах.
4. Утвержденный стек не меняем без отдельного обоснования.
5. Бизнес-правила не дублируем во frontend: логика остается в backend/domain слое.

## 2. Подтвержденные продуктовые решения

Зафиксировано по согласованию:

1. Главные пользовательские блоки Mini App: `Записаться`, `Мои заявки`, `Профиль`, `Поддержка`, `Уведомления`.
2. Админ-часть делаем визуально более строгой.
3. Пользовательскую часть стилизуем по референсу, адаптируя под продуктовый UX.
4. Тексты интерфейса переписываются в более продуктовый UX-стиль.
5. Кастомная графика для MVP: только иконки + 1-2 иллюстрации ключевых экранов.
6. Дедлайн первого релиза в `dev`: 31.05.2026.

## 3. Границы реализации (reuse vs new)

### 3.1 Что переиспользуем из текущего backend

1. Доменные правила расписания: `app/domain/scheduling.py`.
2. Жизненный цикл заявок: `app/domain/lifecycle.py`.
3. Доменные исключения: `app/domain/exceptions.py`.
4. Репозитории и модели БД: `app/db/repositories.py`, `app/db/models.py`, `app/db/enums.py`.
5. Пользовательский сервис сценария: `app/bot/user_flow_service.py`.
6. Админ-сервис и Google OAuth: `app/bot/admin_service.py`.
7. Google Calendar интеграция: `app/services/google_calendar.py`.
8. Фоновые задачи/напоминания: `app/services/background_jobs.py`.

### 3.2 Что добавляем новым для Mini App

1. Отдельный модуль Mini App (`app/miniapp/*`) с web/API-слоем.
2. Frontend статику Mini App (`app/miniapp/static/*`).
3. Авторизацию через Telegram WebApp `initData` (и dev-login только для dev-режима).
4. API-контракты Mini App для user/admin сценариев.
5. Тесты Mini App (`tests/test_miniapp_*`).

### 3.3 Что не должно происходить

1. Замена или удаление текущих Telegram handlers.
2. Перенос бизнес-правил во frontend.
3. Ломающее изменение существующих bot-flow сценариев.

## 4. Архитектура Mini App

### 4.1 Frontend

1. Telegram Mini App frontend на статике (`index.html`, `styles.css`, `app.js`).
2. Два визуальных контура:
   - user: более “живой” стиль;
   - admin: более строгий стиль.
3. Все операции идут через backend API; frontend не принимает бизнес-решений.

### 4.2 Backend API

Базовый префикс: `/api/miniapp`.

Группы API:

1. Auth:
   - `POST /auth/telegram`
   - `POST /auth/dev-login` (только при включенном dev-флаге)
   - `POST /auth/logout`
2. Profile & dashboard:
   - `GET /me`
   - `GET /dashboard`
   - `GET /support`
   - `GET /notifications`
3. Booking:
   - `GET /booking/config`
   - `GET /booking/week`
   - `GET /booking/slots`
4. Requests:
   - `POST /requests`
   - `GET /requests`
   - `PATCH /requests/{request_id}/goal`
   - `POST /requests/{request_id}/cancel`
   - `POST /user/data-deletion`
5. Admin:
   - `GET /admin/requests`
   - `POST /admin/requests/{request_id}/approve`
   - `POST /admin/requests/{request_id}/reject`
   - `POST /admin/requests/{request_id}/alternative`
   - `GET /admin/requests/{request_id}/history`
   - `POST /admin/requests/{request_id}/block`
   - `POST /admin/requests/{request_id}/unblock`
   - `POST /admin/requests/{request_id}/manual-create`
   - `GET /admin/settings`
   - `PATCH /admin/settings`
   - `GET /admin/google/oauth/url`
   - `POST /admin/google/oauth/exchange`

Web route Mini App:

1. `GET /miniapp` — UI.
2. `GET /api/miniapp/health` — health Mini App API.

### 4.3 Авторизация Telegram WebApp

1. Основной режим: верификация `initData` (HMAC + контроль `auth_date`).
2. После успешной валидации создается backend-session token с TTL.
3. Роль определяется на backend (`admin`/`user`), не доверяется frontend.
4. Dev-авторизация разрешается только через `MINIAPP_DEV_LOGIN_ENABLED=true` в `dev`.

### 4.4 Совместимость с текущим ботом

1. Mini App подключается как дополнительный канал, а не как замена.
2. Feature flags (`MINIAPP_ENABLED`, `MINIAPP_DEV_LOGIN_ENABLED`) позволяют безопасно включать/выключать Mini App.
3. Bot-flow и Mini App используют общие доменные сервисы и одни данные в БД.

## 5. User-flow Mini App (MVP parity)

### 5.1 Пользователь

1. Авторизация в Mini App.
2. Главный экран: 5 ключевых блоков.
3. Сценарий записи:
   - выбор длительности;
   - выбор недели/даты;
   - выбор свободного слота;
   - заполнение контактов/цели;
   - согласие на обработку данных;
   - создание заявки.
4. Раздел `Мои заявки`:
   - список;
   - изменение цели до согласования;
   - отмена заявки.
5. `Профиль`, `Поддержка`, `Уведомления`.
6. Запрос удаления/анонимизации данных.

### 5.2 Администратор

1. Просмотр очереди заявок.
2. Согласование/отклонение заявки.
3. Предложение альтернативного слота.
4. История статусов заявки.
5. Блокировка/разблокировка пользователя.
6. Ручное создание встречи.
7. Управление настройками расписания.
8. Google OAuth (получить URL, обменять код).

## 6. БД и миграции

Требование для MVP:

1. Максимально использовать уже существующую схему.
2. Новые миграции добавлять только при реальной необходимости для Mini App.
3. Не создавать “параллельные” таблицы под те же сущности заявок/пользователей.
4. Любая миграция должна быть идемпотентно применима в `dev` и на VPS.

На текущем шаге:

1. Критических новых миграций только под Mini App не требуется.
2. Важно контролировать консистентность `alembic_version` в локальных окружениях.

## 7. Безопасность и персональные данные

Минимальные требования MVP:

1. Проверка подлинности Telegram WebApp `initData`.
2. Session TTL и logout.
3. Жесткая role-check авторизация admin endpoints.
4. Dev-login только для `dev`, в prod отключен.
5. Логи без секретов (токены, OAuth credentials, чувствительные данные).
6. Поддержка пользовательского запроса удаления/анонимизации данных.
7. Согласие на обработку ПДн до отправки заявки.

## 9. Риски и план минимизации

1. Риск: регрессия в текущем Telegram-боте.  
   Митигация: reuse domain слоя + smoke bot-flow после каждого этапа.
2. Риск: рассинхрон логики между bot и Mini App.  
   Митигация: логика только в общих сервисах backend/domain.
3. Риск: ошибки авторизации Mini App.  
   Митигация: строгая валидация `initData`, TTL, role checks, negative tests.
4. Риск: проблемы миграций на окружениях.  
   Митигация: контроль `alembic_version`, прогон миграций в `dev` и на сервере.
5. Риск: конфликт параллельных инстансов polling.  
   Митигация: в локальной проверке Mini App использовать `TELEGRAM_POLLING_ENABLED=false`.
